Writing good C++14

Within C++ there is a smaller, simpler and safer language waiting to get out
  1. Initial work on making CPP better:
    • Core guidelines for CPP
    • Guidelines support library
    • Analysis tool by Microsoft (https://docs.microsoft.com/en-us/cpp/code-quality/code-analysis-for-c-cpp-overview?view=vs-2019)
  2. Additional talks:
    • CppCon 2015: Herb Sutter "Writing Good C++14... By Default"
    • Some more (maybe later)
  3. A set of guidelines to write CPP with:
    1. High-level rules:
      • Express ideas directly in code (compilers don't write comments)
      • Write ISO Standard C++ (https://isocpp.org/std/the-standard)
      • Express intent
      • Ideally a program should be statically type-safe
      • Prefer compile-time checking to runtime checking
      • What cannot be checked at compile time should be checked at runtime
      • Catch runtime errors as early as possible
      • Don't leak any resources
      • Don't waste time or space
    2. Low-level rules:
      • Used primarily in tooling
      • Derived from standards
      • Help to unify style
      • Not minimal or orthogonal
    3. Guidelines Support Library
  4. Solving dangling pointers:
    • Distinguish between owners and non-owners
    • Assume raw pointers to be non-owners
    • Every object has an owner (RAII), use smart pointers
    • Use GSL owner<T> construct (which is just an alias for T*) only for naked pointers!!!
  5. The owners need to disappear into abstractions
